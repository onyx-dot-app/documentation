Skills are reusable instruction bundles that turn a one-off prompt into a repeatable capability, like building a
branded deck or pulling a weekly report the same way every time. An organization skill runs on shared work and
can carry scripts the agent follows, so publishing one is a governance decision and keeping what's live audited
is an ongoing job.

Manage organization skills from **`/craft/v1/skills/manage`**.

## Skill types

| Type | Owner | Visibility |
| --- | --- | --- |
| Built-in | Onyx | On when enabled and dependencies are met. |
| Organization | Admins | Shared with everyone or with selected groups. |
| Personal | Individual users | Visible only to the uploader. |

Built-in skills are company search, slide generation, and image generation. Some need a configured provider
(image generation needs an image model) and appear only once it's in place. The exact set changes as Onyx adds
or removes built-in capabilities.

## Anatomy of a skill bundle

A custom skill is a zip with `SKILL.md` at the root. The frontmatter needs a non-empty `name` and `description`,
and the zip filename becomes the skill slug, so use stable, lowercase, hyphenated names.

```text
weekly-sales-report.zip
  SKILL.md
  examples/
    output.md
  templates/
    report.md
```

| Limit | Default |
| --- | --- |
| Per-file size in a bundle | 25 MB |
| Total bundle size | 100 MB |
| Personal skills per user | 50 |

A bundle is rejected if it omits root `SKILL.md`, has invalid frontmatter, includes unsafe paths, collides with
a built-in slug, or exceeds a limit. Limits are operator-configurable on self-hosted deployments.

## Publish an organization skill

<Steps>
  <Step title="Inspect the bundle">
    Read `SKILL.md` and every example, template, script, and included file. Confirm it has a clear purpose and
    passes the [review checklist](#review-before-sharing).
  </Step>

  <Step title="Upload and share">
    Upload the zip, then share it with everyone or with selected groups.
  </Step>

  <Step title="Test as a pilot user">
    Start a session as someone who should have access and confirm the skill behaves as expected.
  </Step>
</Steps>

### Review before sharing

Confirm the bundle:

- Contains no secrets, tokens, or production credentials.
- Gives clear instructions without overriding organization policy.
- Reaches external services only through configured [apps](/admins/managing_features/craft_apps).
- Tells Craft when to ask before sensitive or irreversible actions.
- Includes examples that match the output quality you expect.

## Govern over time

Enable, disable, replace, or delete organization skills at any time.

<Note>
  Changes take effect on the next session or turn. A running sandbox keeps its current skill set until it
  restarts or the user starts a new session, so availability and group changes won't appear mid-session.
</Note>

Treat the skill library as something you maintain, not a set-and-forget list:

- **Know what's live.** Confirm each shared skill still has a clear owner and purpose.
- **Re-audit on a cadence.** Re-read published bundles and retire ones that no longer match how the work is done.
- **Re-check custom bundles** for secrets, policy overrides, and external calls that bypass configured apps.
- **Account for personal skills.** They're user-owned and outside org review, so use apps and action policies to
  bound what any skill can do.

Users can see how skills surface in a session in [Skills and Apps](/overview/core_features/craft_skills_and_apps).
